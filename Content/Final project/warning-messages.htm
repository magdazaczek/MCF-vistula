Warning messages

The purpose of warning messages

Warning messages aim to highlight an issue and help users avoid potential mistakes. They are different from error messages in that they are displayed before a user takes action. The message should alert the user about potential risks or issues that need attention. Unlike error messages, which stop users from continuing due to an issue, warning messages warn users and encourage them to reconsider an action that might lead to unwanted outcomes. Very often, users can still proceed if they choose to.

Use warning messages for the following situations:

  • To draw attention to something even if taking an action isn’t necessary or required.

  • To indicate that something should be fixed to proceed to the next step without interruptions.

  • To remind users of an upcoming deadline, expiration, or pending request that could cause future issues if not addressed now.

  • To warn users that changing configuration or settings might lead to unwanted outcomes.

Similarly to error messages, warnings shouldn’t include any jargon or phrases that users won’t be familiar with. Keep messages as simple as possible to avoid making users more frustrated.

Examples of warning messages

A good warning message helps users avoid actions that could cause data loss or irreversible changes. However, it’s important to keep a balance in using warning messages and consider if a selected action may result in some negative consequences and it needs to be highlighted. As always, you need to remember that users don’t want to be bombarded with many UI messages and feel overwhelmed by getting them.

Recommendations and best practices

Warning messages shouldn’t be overused and displayed too often to users. If you’re creating warning messages for trivial things, there’s a risk that people will start ignoring all warnings they see in an application. You need to recognize what actions can potentially result in unwanted outcomes. Users need to see warning messages before an action so that they’re informed of what will happen if the action is taken. Warning messages should use language that feels natural and human, even when the message is serious.

The table below presents recommendations and best practices for the key aspects of warning messages.

Warning messages Recommendations and best practices
Title

  • Use titles only when necessary. Avoid using “Alert” or “Warning” as the only word in the title. It’s not helpful at all.

  • Avoid explaining what to do in titles.

  • Avoid using exclamations.
Body

  • Include a reason for the warning and display the message before the user takes an action.

  • Avoid repeating text from the title.

  • Don’t write more than 2 sentences.
Call to action (CTA)

  • Be specific. If a warning message invokes a choice, use imperative verbs such as “Save” or “Create”.

  • Use as few words for CTA as possible.
Accessibility

  • Don’t use jargon.

  • If there are any links needed, make them as descriptive as possible.

  • Make text easily scannable to focus the user’s attention on key information.
